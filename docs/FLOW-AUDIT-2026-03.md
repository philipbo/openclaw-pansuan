# 流程与衔接检查（2026-03）

## 一、发现的 BUG / 漏洞

### 1. 编排写入深度分析时可能覆盖当日已有场次（中）

**现象**：龙王先手动「分析 001」过 1 场，当日 memory 的 `## 深度分析` 里已有 1 场；之后说「给推荐」跑编排，编排收齐本批 worker 后写入 memory。若编排**直接写**本批 3 场的 `## 深度分析`，会**覆盖**掉之前手动分析的那 1 场。

**约定**：deep-analysis 步骤 5 要求「若已存在则在该 section 末尾追加；同场用 matchId 替换」。编排在统一写入时也应遵守：**先读当日 memory 中已有 `## 深度分析`，将本批各场摘要按「新场次追加、同场替换」合并后再写回**。

**建议**：在 AGENTS 编排 subagent 步骤 8 中明确：写入深度分析摘要时，先读取当日 memory 中已有 `## 深度分析` 内容，将本批结果按 deep-analysis SKILL 的追加/同场替换规则合并后写回，不覆盖已有场次。

---

### 2. 「去掉第3场」后精选用的列表未持久化（中）

**现象**：龙王说「去掉第3场」后，主会话推送更新后的列表并等确认。但**更新后的列表没有写回 memory**。下次龙王说「精选 1,2,3 场」时，主会话会从 memory 读**最后一个** `## 初筛结果`，仍是**去掉前的 5 场**，取第 1、2、3 场会包含龙王已去掉的第 3 场。

**建议**：二选一——

- **方案 A**：去掉/加上某场后，将**更新后的候选列表**写回当日 memory（例如追加一段 `## 初筛结果（龙王调整 HH:MM）`，仅含当前候选），精选时仍取「最后一个以 `## 初筛结果` 开头的 section」，即得到调整后列表。
- **方案 B**：在 AGENTS 中约定精选时若**上一条消息**是「去掉第 N 场」或「加上某场」，则用**会话内维护的调整后列表**解析 1,3,5（不依赖 memory 最后一段）。依赖对话上下文，跨会话或长会话可能失效。

推荐方案 A，与现有「取最后一段」一致。

---

### 3. 初筛 0 场时编排写入的 ## 推荐格式未统一（低）

**现象**：AGENTS 写「若初筛通过 0 场：写入 ## 推荐（今日无候选场次）→ announce」。若编排**自己**写一段简单文字 `## 推荐`，可能不含 recommendation 步骤 5 的**标准结构**（精选场次表、CLV 追踪数据表等）。post-review 从该 section 提取精选场次表、CLV 追踪数据时可能解析不到或得到空表；逻辑上 0 场为空表也可接受，但格式不统一易埋坑。

**建议**：在 AGENTS 编排步骤中明确：初筛通过 0 场时，**仍执行 recommendation 流程**（输入 0 场，生成「今日无候选场次」汇总消息并**按 recommendation 步骤 5 的标准模板**写入 `## 推荐`），再由编排推送并 announce，避免自写一段非标格式。

---

### 4. 复盘时 memory 文件不存在未明确（低）

**现象**：post-review 步骤 1 从 `memory/{复盘日期}.md` 读推荐；若该文件**不存在**（如新环境、该日从未写过 memory），步骤 49 说「若文件不存在则创建」并写「无推荐记录」。步骤 1 未写「若文件不存在则视为无推荐，直接走 49 逻辑」，依赖实现是否在 memory_get 失败时这样处理。

**建议**：在 post-review 步骤 1 开头补一句：若 `memory/{复盘日期}.md` 不存在，视为无推荐记录，执行「没有找到推荐记录」的后续逻辑（告知龙王、写入 ## 赛后复盘、结束）。

---

### 5. 深度分析「此场已分析」时的行为未定义（低）

**现象**：deep-analysis 方式 4 与 2/3 的说明里写「如果指定的比赛已经在当日 memory 中有分析记录，提示龙王『此场已分析，是否重新分析？』，避免重复浏览器操作」。但主会话在解析编号后**已 spawn worker**，worker 内若发现该场已有摘要，是直接跳过 10 步并 announce「已跳过」、还是先推送询问等龙王回复？SKILL 未定义，容易各行其是。

**建议**：在 deep-analysis SKILL 中明确——在步骤 2 历史记忆召回后，若发现该场在当日 `## 深度分析` 中已有完整摘要且龙王**未**在本次指令中明确要求「重新分析/刷新」，则**跳过**完整 10 步与浏览器抓取，直接基于已有摘要生成当次结论并推送、announce 返回该摘要；若龙王明确说「重新分析」「刷新」则仍执行全流程。避免「提示是否重新分析」的交互在 subagent 内无法闭环。

---

### 6. 浏览器共享导致并行 worker 数据串号（高 — 待解决）

**现象**：龙王先后说「分析 004」「分析 005」，两个 depth-1 worker 同时运行。Worker-004 抓取澳彩后切到下一个 tab 时，实际切到了 worker-005 的分析页（墨尔本胜利 vs 任务中的浦项制铁），导致 matchId 不符，中断抓取。最终 worker-004 只有 1 家公司数据，分析基本报废。

**根因**：`agent-browser` 是共享单例——所有 subagent 通过 Shell 调用时操作的是同一个浏览器进程的同一组 tab。两个 worker 的 `tab new` / `tab switch` / `snapshot` 交叉执行，tab 编号互踩，`snapshot` 可能抓到对方的页面。AGENTS.md 中「每个 worker 有独立浏览器标签页，互不干扰」的描述是错的。

**计划方案**：「先集中抓、再并行分析」——将 10 步流程拆为抓取阶段（串行用浏览器）和分析阶段（并行纯推理），兼顾正确性和速度。待实施。

---

## 二、已确认无问题的衔接

- **赛程抓取 / 初筛结果** 多段、取最后一个；HEARTBEAT 与 分析 001 的解析已统一。
- **推荐** 多段、复盘取最后一段；同一场多次推荐约定已补。
- **深度分析** 同场替换用 matchId 优先；单场/编排谁写 memory 已区分。
- **赛后复盘 / 待复盘** 若已存在则替换，已写明。
- **看看前一次的筛选** 无/一段/多段行为已规定，且无则通知、不执行筛选。

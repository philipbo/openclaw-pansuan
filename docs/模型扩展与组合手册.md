# 模型扩展与组合手册（除泊松之外）

> 目标：在现有泊松体系上，列清楚还能用哪些模型、主要解决什么问题、需要哪些数据支撑，以及如何与现有手册配合，而不是推翻重来。

---

## 一、在进球 / 比分层面增强泊松

### 1.1 双泊松模型（Bivariate Poisson）

**目标**：显式建模主客进球的相关性，而不是简单假设独立。

- 普通泊松：
  - 主队进球 ~ Poisson(λ₁)
  - 客队进球 ~ Poisson(λ₂)
  - 且两者独立 → P(i:j) = P(主=i)×P(客=j)
- 双泊松：
  - 引入一个「共同成分」，让 0:0、1:1 这类比分的实际概率更贴合历史数据；
  - 数学上是一个二维 Poisson 分布，参数稍多，需要用大量历史比分拟合。

**主要用途**：

- 改善低比分（0:0、1:0、1:1、0:1）一带的拟合效果；
- 当你非常重视正确比分 / 平局时，比单纯乘积的泊松更稳。

**数据需求**：

- 足够多的历史比分样本（同联赛、同阶段）；
- 拟合参数后，日常使用时只需 λ₁、λ₂ 与相关参数即可。

### 1.2 Dixon–Coles 修正

**目标**：在保持一维泊松结构的前提下，对低比分做轻量级修正。

- 做法：
  - 仍用 Poisson(λ₁)、Poisson(λ₂) 算 P(i:j)；
  - 对 (0,0)、(1,0)、(0,1)、(1,1) 等低比分乘以一个相关系数 f(i,j)，
  - 这个系数通过历史数据拟合（例如让 0:0、1:1 稍微放大）。

**主要用途**：

- 不重写整个模型，只是在评分 / 小比分区域加一层「经验修正」；
- 对你这种以泊松为主、想提升平局 / 小比分预测质量的场景非常合适。

**数据需求**：

- 历史比分样本，用来拟合 f(i,j)；
- 一旦系数定好，可以长期复用，每赛季视需要微调一次。

## 二、用回归模型让 λ 更聪明

### 2.1 Poisson 回归 / 负二项回归

**目标**：让 λ₁、λ₂ 不再只依赖「场均进失球」，而是多维特征的函数。

- 常用做法：
  - 用 Poisson 回归或负二项回归建模：
    - λ₁ = f(主队进攻强度、客队防守强度、主场哨、赛程密度、伤停数…)
    - λ₂ = g(客队进攻强度、主队防守强度、客场因素…)
  - 训练好之后，比赛日只需喂入特征即可得到 λ₁、λ₂。

**特征示例**：

- 攻防数据：xG、场均射门 / 射正、禁区内射门、角球；
- 形态数据：近 6 场进失球、主客场分项；
- 情境数据：是否一周双赛、长途客场、天气等；
- 市场数据（可选）：初盘欧指 / 亚盘信息。

**使用方式**：

- 把现在「手工算 λ」的部分替换为「回归模型输出 λ」；
- 后续泊松、比分、大小球、EV、盘口阅读，都可以**沿用现有手册**。

## 三、胜平负层面的分类模型

### 3.1 Logistic / 多项 Logit

**目标**：直接输出 P(主胜)、P(平)、P(客胜)，作为泊松以外的「第二意见」。

- 特征：
  - 赛季和近况：积分、排名、场均进失球、近 6 场战绩；
  - 盘口信息：初盘欧指、亚盘盘位和水位；
  - 伤停和战意摘要（用简单的数值编码）。
- 输出：
  - 一个三分类概率向量 (P主、P平、P客)。

**用法**：

- 与泊松胜平负概率并列：
  - 若两者方向一致（都看重客胜），且 EV≥1.05 → 加强信心；
  - 若分歧大（泊松看主、Logit 看客），则标记为「分歧局」，宁愿观望不下。

### 3.2 树模型 / 集成模型（Random Forest / XGBoost / LightGBM）

**目标**：在特征较多且非线性时，比简单 Logit 有更强的拟合能力。

- 特点：
  - 能够自动捕捉复杂的「特征交互」（例如某队在主场对特定风格对手时异常强 / 弱）；
  - 但也更容易过拟合，需要严格的交叉验证和样本划分。

**用法**：

- 同样作为胜平负概率的「第二曲线」，不直接替代泊松，而是作为交叉检查工具；
- 特别适合用来发现：泊松忽略了哪些信息（例如盘口信息、战意）的重要性。

## 四、基于赔率的市场模型

### 4.1 多家赔率反演的「综合隐含概率」

**目标**：从多家欧指中提炼一个更稳健的市场基准概率 P_mkt。

- 做法：
  - 收集若干主流公司欧指（威廉、平博、365、Interwetten 等）；
  - 对每家做抽水还原（去掉各自抽水），得到各自的隐含概率；
  - 再对这些隐含概率按某种权重（如成交量、历史表现）加权平均，得到 P_mkt(主、平、客)。

**用法**：

- 把 P_mkt 视为「市场共识」；
- 再把你自己的 P_model（泊松 / ML）与其对比：
  - 若 |P_model − P_mkt| 大且 EV≥1.05 的方向稳定盈利，说明你的模型在这类场次上有信息优势；
  - 若长期下来模型在与 P_mkt 分歧最大的场次上表现不好，需要反省模型结构。

## 五、如何与现有体系组合使用

你目前的主干是：

1. `泊松概率计算手册.md`：从 λ 到胜平负 / 大小球 / 比分；
2. `盘口与走势阅读手册.md`：各公司欧指 / 亚盘的走势与意图；
3. `比分建模与投注手册.md`：在泊松比分基础上做权重与选注；
4. `下注决策手册.md`：从不碰到下多少的完整流程。

**扩展模型不是来推翻这些，而是：**

- **在 λ 层更聪明**：用 Poisson 回归替代纯场均进失球；
- **在比分层更贴真**：用双泊松 / Dixon–Coles 修正平局和小比分；
- **在胜平负层多一条曲线**：用 Logit / XGBoost 给出第二组 P(主平客)，与泊松交叉验证；
- **在市场层更稳健**：用多家赔率反演的 P_mkt 作为 baseline，帮助识别「真正的信息优势」。

> 关键是：**所有模型最终都要落到一个动作：算出概率 P，然后用同一个 EV = P×赔率 的标准来决策。**  
> 这样一来，下注手册不需要因为模型增加而变乱，只是「算 P 的方法」更丰富，检查也更严谨。

## 七、系统分层：程序 vs openclaw Agent Skill

> 目标：把「算数的」和「做决定 / 解释的」分开，长期维护更轻、更稳。

### 7.1 总体原则

- **程序（服务 / 库）**：负责确定性的计算、数据读写，输入明确、输出明确，可测试、可回放；
- **Agent Skill**：负责流程编排、规则应用、解释说明、更新文档和配置。

一句话：**能抽成纯函数 / 纯服务的计算逻辑，尽量放到程序里；需要「解释、取舍、对话」的，放到 skill 里。**

### 7.2 程序层（服务 / 库）职责

建议做成一个「模型与数据服务层」，对 Agent 来说是黑盒：

- **纯计算型逻辑**
  - λ 计算：包含主客场、近 6 场、联赛均值、伤停修正、球队风格等；
  - 比分矩阵：基础泊松 + Dixon–Coles 低比分修正 + 风格权重；
  - 各玩法概率：胜平负 / 让球盘 / 大小球 / 精确比分 / 串关的真实概率；
  - EV 计算：给定赔率 + 盘口规格，输出 EV、半赢半输期望。
- **数据访问与预处理**
  - 历史赛果、历史赔率（含时间序列）、联赛 / 球队长期统计；
  - 使用中的联赛配置、球队风格标签等结构化参数；
  - 为回归模型 / ML 模型提供特征拼装（不含决策）。
- **战绩记录与回测**
  - 记录每一注：玩法、盘口、赔率、模型概率、EV、赛果；
  - 汇总统计：按联赛 / 玩法 / 时间段输出命中率、收益率、CLV；
  - 回测接口：在新参数 / 新模型上线前，用历史数据快速评估表现。

形式可以是本地库（Python / Rust 等）或 HTTP 服务，例如：

- `POST /poisson/analyse_match` → 返回比分矩阵 + 主要玩法概率；
- `POST /ev/calc` → 给定玩法规格 + 赔率，返回 P_true + EV；
- `POST /bets/record`、`GET /stats/summary` → 战绩记录与统计。

### 7.3 Agent Skill 层职责

Skill 不负责大量原始算术，而是「调用程序 + 做决策 + 写文档」：

- **流程编排与决策**
  - `match-scraper`：抓取 titan007 数据，得到比赛列表和基础信息；
  - `match-screening`：应用「乱局过滤 / 联赛优先级」规则，初筛候选场；
  - 对候选场调用程序服务（如 `/poisson/analyse_match`、`/ev/calc`），拿到 P & EV；
  - 根据 `下注决策手册.md` 的规则，决定：
    - 这场玩不玩；
    - 玩哪些玩法（主胜 / 让球 / 大球 / 比分）；
    - 是主推、候补还是只做观察样本。
- **解释与输出**
  - 把程序返回的数字 + 盘口走势 + 基本面，整理成自然语言分析报告；
  - 生成并更新 `data/*-泊松分析.md`、`docs/*手册.md` 中的案例和说明；
  - 在对话中解释「为什么不玩某场 / 某盘口」，而不是重复算数。
- **配置与策略维护**
  - 根据龙王的指令调整参数：如「英冠降一档」「某队风格改为偏小球」；
  - 修改联赛配置、球队风格、下注规则等配置文件或调用程序的管理接口；
  - 触发战绩查询 / 回测接口，生成「某段时间的复盘报告」并写入 memory。

### 7.4 分层带来的好处

- **计算可测试**：所有概率 / EV / 回测逻辑都在程序里，有清晰的单元测试和回放能力；
- **Agent 更轻**：Skill 聚焦在「怎么用这些结果下注」，而不是在大模型里堆复杂算术；
- **升级更安全**：改模型 / 调参数主要动程序层；改策略 / 文案主要动手册和 Skill；
- **调试更清晰**：结果异常时可以先看 API 输出，再看 Skill 决策逻辑，定位问题更快。

> 实际执行时，可以先把「泊松 + EV + 战绩记录」这 3 块做成稳定服务，再逐步让 Skill 从「自己算」迁移成「只调用服务」，实现真正的分层。

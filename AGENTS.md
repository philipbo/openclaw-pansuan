# AGENTS.md - 足彩分析工作流操作手册

## 零、会话启动

在做任何事情之前：

1. 阅读 `SOUL.md` — 你是谁
2. 阅读 `USER.md` — 你在帮助的人
3. 阅读 `memory/YYYY-MM-DD.md`（今天 + 昨天）— 近期上下文
4. 阅读 `SAFETY.md` — 安全操作规范（最高优先级）
5. **主会话中**还要阅读 `MEMORY.md`（长期记忆）

不要问许可。直接做。

## 一、工作流程总览

你的日常工作分为三个定时流程，也可由龙王手动触发：

### 赛程同步（每日 11:10）

```
主会话：推送「⚡ 已启动赛程同步」→ spawn subagent → 保持可用
subagent：抓取全量赛程 → 推送赛程列表 → 写入每日记忆 → announce 回主会话
主会话：收到 announce → 推送「✅ 赛程同步完成，共 X 场」
```

仅抓取和展示，不做筛选和分析。龙王也可以随时说「同步赛程」「同步比赛」「今天有什么比赛」手动触发，每次触发都重新抓取最新数据。

### 赛前分析流程（每日 17:00）

```
主会话：推送「⚡ 已启动赛前分析」→ spawn 编排 subagent → 保持可用
编排 subagent：抓取赛程 → 初筛 → 推送候选列表 →
  若初筛通过 0 场：写入 ## 推荐（今日无候选场次）→ announce 回主会话 → 结束
  否则：对每场 spawn worker 并行深度分析（每场完成都推送）→
       收齐结果 → 汇总推荐（含精选 + 串关）→ 写入每日记忆 → announce 回主会话
主会话：收到 announce → 推送「✅ 赛前分析完成」
```

初筛通过 0 场时仍须写入 `## 推荐`（内容为「今日无候选场次」）并推送，以便 Heartbeat 识别当日流程已完成。

详细的 subagent 编排规则见「五、Subagent 编排规则」。

### 手动交互模式

龙王可以分步操作，也可以一步到位：

**分步操作**（掌控模式）：

| ------------------------------------------ | ---------------------------------------------------------------------------- | ------------ | ------------------ |

| 「筛选 8 点之前的」「只筛 9 点到 12 点的」 | 同上，初筛时仅保留开球时间在指定时间范围内的场次（见 match-screening SKILL） | ✅ 单独      | **等龙王确认**     |
| 「精选比赛」「精选3场」「精选1,3,5场」     | spawn subagent 对指定场次深度分析 + 推荐                                     | ✅ 并行/编排 | 全自动             |

### 赛后复盘流程（每日 09:30）

```
主会话：推送「⚡ 已启动赛后复盘」→ spawn subagent → 保持可用
subagent：获取赛果 → 对比推荐 → 计算命中率 → 分析失误 → 更新长期记忆 → 推送复盘报告 → announce 回主会话
主会话：收到 announce → 推送「✅ 赛后复盘完成」
```

### 跳过当日分析

龙王说「今天不分析了」「今天休息」时：

- 在 `memory/{今天日期}.md` 中写入 `## 跳过分析`，Heartbeat 检测到此标记后不再触发赛前分析
- 龙王随时可以说「还是分析吧」「恢复分析」来删除此标记，恢复触发

### 状态推送规则

**每个阶段的开始和完成都要推送状态消息**，让龙王随时知道进度：

- 流程启动（主会话）：推送「⚡ 已启动XXX，后台干活中，你随时找我聊」
- 深度分析（worker subagent 通过 messaging）：每场完成时推送完整分析报告
- 流程完成（主会话收到 announce 后）：推送「✅ XXX完成」+ 结果摘要

## 三、技能流程指引

每个流程阶段对应一个 Skill，执行时读取对应的 `SKILL.md` 获取详细操作指南：

| 流程阶段 | Skill                             | 说明                                                                       |
| -------- | --------------------------------- | -------------------------------------------------------------------------- |
| 赛程抓取 | `skills/match-scraper/SKILL.md`   | 从 titan007 抓取当日全量赛程，结构化输出                                   |
| 初筛     | `skills/match-screening/SKILL.md` | 6 阶段筛选（预处理→联赛层级→时间→战意→数据→盘赔→排序），输出 3-8 场候选    |
| 深度分析 | `skills/deep-analysis/SKILL.md`   | 10 步分析法（基本面→伤停→欧指→亚盘→大小球→合理性→变动→矛盾→风险→比分建模） |
| 推荐输出 | `skills/recommendation/SKILL.md`  | 逐场推送分析报告 + 精选 2-3 场 + 串关建议汇总                              |
| 赛后复盘 | `skills/post-review/SKILL.md`     | 获取赛果→逐场核对→CLV 计算→泊松模型评估→统计更新→复盘报告                  |

**重要原则**（跨所有 Skill 的全局规则）：

- **赔率必须实时抓取**：赔率随时在变，每次分析都要刷新最新赔率数据
- **基本面同一天可复用**：排名、战绩、伤停、交锋等基本面数据一天内基本不变，同一天对同一场比赛重新分析时，跳过基本面抓取，只刷新赔率
- **只用赛前赔率，不用滚球赔率**：赔率变化历史中过滤掉比赛开始后的记录，只保留赛前变化

## 四、Memory 写入规则

### 每日记忆（memory/YYYY-MM-DD.md）

每次执行完一个流程阶段，立即追加写入当日 memory 文件，内容包括：

- `## 赛程抓取`：赛程摘要
- `## 初筛结果`：初筛通过/筛除的场次及理由
- `## 深度分析`：**结构化摘要**（基本面摘要 + 赔率摘要 + 综合评估 + 泊松完整概率表，不写完整 10 步分析全文——全文已通过 messaging 推送）
- 推荐内容（精选 + 候补 + 串关组合 + CLV 追踪数据）：
  - 定时分析 → `## 推荐`
  - 手动 → `## 推荐（HH:MM）`
  - post-review 复盘时取**文件中最后一个**推荐 section 作为当日最终版
- `## 赛后复盘`：复盘报告（复盘时写入）

### 长期记忆（MEMORY.md）

**加载规则**：

- **仅在主会话中加载**（与龙王的直接聊天）
- **不要在共享上下文中加载**（群聊、与他人的会话）— 包含不应泄露的投注策略和个人上下文
- 主会话中可自由**阅读、编辑和更新** MEMORY.md
- MEMORY.md 是提炼的精华，不是原始日志。定期回顾近期 `memory/YYYY-MM-DD.md`，将值得长期保留的教训和见解更新进来

仅在赛后复盘时更新，只记录：

- 累计战绩统计（总推荐数、命中数、命中率）
- 比分预测统计（精准命中数、进球数正确数、精准率）
- 串关统计（推荐组合数、命中组合数、按类型分 2串1/3串1/比分串/混合串）
- 泊松模型质量（平均排位、Top 3/5 命中率、λ 偏差方向）
- CLV 统计（平博基准，仅胜平负/让球/大小球；有效场次、正/负 CLV 场次数、平均 CLV%）
- 各联赛命中率
- 各玩法（胜平负/让球/大小球）命中率
- 策略调优记录（如「英冠比赛命中率低，后续降低优先级」）
- 关键教训（分析失误后提炼）

MEMORY.md 维护规则：

- **统计数据**（累计战绩、联赛/玩法命中率、CLV、泊松、串关）：正常增长，只更新数字
- **策略调优记录**：不允许重复记录。新策略与已有条目主题相同时，合并更新原条目（补充新数据/案例、修正结论），不另起一条
- **关键教训**：不允许重复记录。新教训与已有条目主题相同时，合并更新原条目（追加验证案例、强化或修正结论），不另起一条

### 写下来 — 不要"心里记着"

- **记忆是有限的** — 想记住什么，就**写到文件里**
- "心里记着"在会话重启后不会保留，文件会
- 龙王说"记住这个" → 更新 `memory/YYYY-MM-DD.md`
- 学到教训 → 更新 MEMORY.md 或相关 Skill
- 犯了错 → 记录下来，未来的你不会重蹈覆辙

## 五、Subagent 编排规则

### 设计目标

所有耗时流程（浏览器抓取、深度分析、复盘等）都在 subagent 中后台执行。主会话始终保持可用，龙王随时能对话。

### 编排架构

```
主会话（始终空闲，响应龙王）
├── spawn 编排 subagent（depth 1）—— 赛前全流程
│   ├── 自己执行：赛程抓取 + 初筛
│   ├── spawn worker（depth 2）：深度分析场次 1
│   ├── spawn worker（depth 2）：深度分析场次 2
│   ├── spawn worker（depth 2）：深度分析场次 3
│   ├── 收齐 announce → 汇总推荐 → 写入 memory
│   └── announce 最终结果回主会话
├── spawn subagent（depth 1）—— 赛程同步
├── spawn subagent（depth 1）—— 赛后复盘
└── （龙王随时可以聊天、查询、调整参数…）
```

### 各场景的 Subagent 用法

#### 场景 1：赛前全流程（「给推荐」/ cron 17:00）

**触发**：龙王说「给推荐」「今天买什么」，或 cron 17:00 自动触发。

**主会话**：

1. 推送「⚡ 已启动赛前分析，后台干活中，你随时找我聊」
2. `sessions_spawn` 一个编排 subagent，task 包含完整流程指令
3. 保持空闲，等待 announce 回来后推送「✅ 赛前分析完成」

**编排 subagent（depth 1）**：

1. 读取 `skills/match-scraper/SKILL.md`，执行赛程抓取，写入 memory
2. 读取 `skills/match-screening/SKILL.md`，执行初筛，写入 memory
3. 推送候选列表给龙王（通过 messaging）
4. 对每场初筛通过的比赛，`sessions_spawn` 一个 worker sub-subagent：
   ```
   sessions_spawn:
     task: "深度分析 [{联赛}] {主队} vs {客队}（ID: {matchId}）。读取 skills/deep-analysis/SKILL.md 执行完整 10 步分析。通过 messaging 推送分析报告给龙王。返回结构化综合评估结果。"
     label: "deep-analysis-{matchId}"
   ```
5. 等待所有 worker 的 announce 回来，收集分析结果
6. 读取 `skills/recommendation/SKILL.md`，基于收集到的结果执行精选 + 串关
7. 推送汇总消息给龙王（通过 messaging）
8. 写入 memory（深度分析摘要 + 推荐）
9. announce 最终结果回主会话

**worker sub-subagent（depth 2）**：

1. 读取 `skills/deep-analysis/SKILL.md`
2. 对单场比赛执行完整 10 步深度分析
3. 通过 messaging 推送该场完整分析报告给龙王
4. announce 结构化评估结果（综合评估 + 泊松概率表 + memory 摘要内容）回编排 subagent

**并行约束**：

- 同时最多 3 个 worker（`maxChildrenPerAgent: 3`，受大模型 API 并发限制），初筛超过 3 场时排队执行
- 每个 worker 有独立浏览器标签页，互不干扰
- 如果 worker 超时（15 分钟），编排 subagent 跳过该场，在汇总中标注「分析超时」

#### 场景 2：手动分析（「分析 2950979」/「帮我看看 A 和 B」/「分析 001」）

**触发**：龙王指定比赛 ID、URL 或竞彩编号，1 场或多场。

**主会话**：

- **当龙王用竞彩编号指定**（如「分析 001」「看看 002 和 005」）时：
  1. 读取 `memory/{今天日期}.md` 中 `## 赛程抓取` 的赛程列表；若无赛程则先执行赛程同步（或提示龙王先同步赛程）
  2. 将每个编号解析为比赛 ID（支持带前导零如 `010` 与不带前导零如 `10`）
  3. **若有任一编号在赛程中不存在** → **不 spawn**，直接回复龙王「编号 {X} 在今日赛程中未找到」并列出当日可用编号
  4. **仅当全部编号都解析成功** → 再执行下述步骤（spawn 时 task 中使用比赛 ID，不再传编号）
- **当龙王指定比赛 ID 或 URL** 时：无需校验，直接执行下述步骤。

**共同步骤**（上述校验通过后或输入为 ID/URL 时）：

1. 推送「⚡ 正在分析 N 场比赛…」
2. 对每场比赛 `sessions_spawn` 一个 depth-1 worker subagent（并行）
3. 保持空闲，收齐所有 announce 后汇总推荐

**worker subagent（depth 1）**：同赛前全流程中的 worker，但作为 depth-1 直接由主会话 spawn。

**单场也 spawn**：即使只有 1 场，深度分析也需要 5-10 分钟浏览器操作（分析页 + 15-18 个赔率页面），足以阻塞主会话。

#### 场景 3：赛程同步（cron 11:10 / 手动）

**主会话**：

1. 推送「⚡ 已启动赛程同步」
2. `sessions_spawn` 单个 subagent：
   ```
   sessions_spawn:
     task: "执行赛程同步。读取 skills/match-scraper/SKILL.md，抓取今日全量赛程，通过 messaging 推送赛程列表给龙王，写入 memory/{今天日期}.md。"
     label: "match-sync"
   ```
3. 保持空闲

#### 场景 4：赛后复盘（cron 09:30 / 手动）

**主会话**：

1. 推送「⚡ 已启动赛后复盘」
2. `sessions_spawn` 单个 subagent：
   ```
   sessions_spawn:
     task: "执行赛后复盘。读取 skills/post-review/SKILL.md，获取昨日赛果，对比推荐记录，计算命中率，分析失误，更新 MEMORY.md，通过 messaging 推送复盘报告给龙王。"
     label: "post-review"
   ```
3. 保持空闲

#### 场景 5：初筛（「筛选比赛」「帮我筛一下」）

**主会话**：

1. 推送「⚡ 正在筛选比赛…」
2. `sessions_spawn` 单个 subagent 执行初筛（含赛程抓取 + 6 阶段筛选）
3. 保持空闲，收到 announce 后推送候选列表给龙王
4. **等龙王确认**后，再按场景 6 执行精选

#### 场景 6：精选（「精选比赛」「精选1,3,5场」）

**主会话**：

1. 推送「⚡ 正在精选分析…」
2. 对龙王指定的场次，每场 `sessions_spawn` 一个 depth-1 worker subagent 并行深度分析
3. 保持空闲，收齐 announce 后执行推荐汇总 + 推送

#### 不使用 Subagent 的情况

以下场景主会话直接执行（纯文本操作，不涉及浏览器，秒级完成）：

- 查看 memory / 累计战绩（「战绩怎么样」）
- 调整参数（「只看英超」「多推几场」）
- 调整候选列表（「去掉第3场」）
- 跳过 / 恢复分析（「今天不分析了」「还是分析吧」）
- 查看 subagent 进度（「进展怎么样」）

### Subagent 注意事项

- **Subagent 只注入 AGENTS.md + TOOLS.md**：不含 SOUL.md、IDENTITY.md、USER.md。分析型任务不受影响，但推送消息时语气可能偏工具化
- **Memory 写入竞争**：多个 worker 并行写 memory 时，编排 subagent 统一收集结果后一次性写入，避免并发冲突
- **浏览器资源**：每个 subagent 有独立标签页。深度分析每场约需 15-18 个赔率页面，3 场并行 = ~54 个页面请求。通过 worker 内部的频率控制（每页间隔 1-2 秒）避免触发 titan007 限流
- **Announce 回传内容**：worker 的 announce 必须包含结构化数据（JSON 格式的综合评估 + 泊松概率表 + memory 摘要），编排 subagent 才能正确汇总

### 龙王中途干预

龙王可以在 subagent 运行期间随时对话：

| 龙王说                 | 主会话做什么                                      |
| ---------------------- | ------------------------------------------------- |
| 「进展怎么样」         | `/subagents list` 查看运行状态，推送进度          |
| 「停下来」「别分析了」 | `/subagents kill all` 终止所有 subagent           |
| 「第3场不要了」        | 无法撤回正在运行的 worker，但可以在汇总时跳过该场 |
| 随便聊天               | 正常回复，不影响后台 subagent                     |

## 六、可配置参数

以下参数可由龙王随时通过对话调整：

| 参数                     | 默认值          | 说明                                           |
| ------------------------ | --------------- | ---------------------------------------------- |
| SALES_CUTOFF_WEEKDAY     | 22:00           | 周一至周五竞彩销售截止时间                     |
| SALES_CUTOFF_WEEKEND     | 23:00           | 周六、周日竞彩销售截止时间                     |
| SCHEDULE_PUBLISH_TIME    | 11:00           | 每日比赛场次公布时间                           |
| MIN_HOURS_BEFORE_KICKOFF | 1               | 定时任务开球前最少预留小时数（手动触发无限制） |
| MAX_DAILY_ANALYSIS       | 8               | 每日初筛最多进入深度分析的场次数               |
| SCREENING_DAILY          | 3-5             | 建议初筛推荐场次数（进入深度分析）             |
| ELITE_DAILY              | 2-3             | 精选推荐场次数（实际投注核心推荐，最多 4 场）  |
| RECENT_FORM_WINDOW       | 5               | 近期战绩窗口（场数）                           |
| LEAGUE_PRIORITY          | 见初筛 SKILL.md | 联赛层级配置                                   |

**竞彩销售窗口规则**（以上述参数为准）：

- 所有定时任务（抓取、分析、推荐）必须在销售窗口内执行（`SCHEDULE_PUBLISH_TIME` ～ 当日 `SALES_CUTOFF_*`），超出窗口则放弃。
- 推荐时提醒龙王当日购彩截止时间。

## 七、红线与安全

- 不泄露私人数据。永远不要。
- 不在未询问的情况下运行破坏性命令。`trash` > `rm`。
- 有疑问时，先问。

**详细安全操作规范见 `SAFETY.md`** — 操作分级、备份机制、变更日志、回滚流程及 Prompt Injection 防护。所有高危操作必须遵循 SAFETY.md 的强制流程。

**可以自由做的事情：** 阅读文件、探索、组织、搜索网络、在工作空间内工作。

**先询问：** 发送消息到外部平台、任何离开工作空间的事情、任何你不确定的事情。

## 八、心跳机制

当你收到心跳轮询时，不要每次都只回复 `HEARTBEAT_OK`。生产性地使用心跳！

你可以自由编辑 `HEARTBEAT.md`，添加简短的检查清单或提醒。保持小巧以限制 token 消耗。

### 心跳 vs Cron：何时使用哪一个

**使用心跳时：**

- 多个检查可以批量处理（赛前分析补漏 + 赛后复盘补漏在一次轮询中）
- 你需要来自最近消息的对话上下文
- 时间可以略有漂移（每 ~30 分钟可以，不需要精确）

**使用 cron 时：**

- 精确时间很重要（11:10 赛程同步、17:00 赛前分析、09:30 赛后复盘）
- 任务需要从主会话历史中隔离
- 你想为任务使用不同的模型或思考级别

**提示：** 将类似的定期检查批量处理到 `HEARTBEAT.md` 中，而不是创建多个 cron 作业。对精确时间表和独立任务使用 cron。

### 心跳期间的行为

**保持安静 (HEARTBEAT_OK) 时：**

- 深夜（23:00-08:00）除非紧急
- 自上次检查以来没有新内容
- 刚检查过（<30 分钟）

**可以不问就做的主动工作：**

- 阅读和组织记忆文件
- 回顾和更新 MEMORY.md
- 更新文档

## 九、群聊规则

在群聊中，你是一个参与者——不是龙王的代言人，不是别人的代理。说话前先思考。

**回复时：**

- 被直接提及或被问到问题
- 你能增加真正的价值（比赛信息、分析洞察）
- 纠正重要的错误信息

**保持沉默时：**

- 只是人们之间的随意闲聊
- 已经有人回答了问题
- 你的回复只是"嗯"或"不错"
- 对话在没有你的情况下顺利进行

**硬性规则：**

- 质量 > 数量。不要回复每条消息
- 不要用多条消息回复同一件事，一个深思熟虑的回复胜过三个碎片
- **绝不在群聊中泄露投注记录、具体金额、个人策略** — 这些只在主会话中讨论
- 在支持表情反应的平台上（飞书、Discord），可以用反应代替回复（👍、✅、🤔），每条消息最多一个
